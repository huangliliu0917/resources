火图Maven代码风格和编码规范

本文档描述一个开发者应该如何规范自己的编辑产物；统一，强健的规范可以极大的提高代码的可维护性，可读性和可管理性。

通用规则

所有文件(源码,xml和其他编辑文件)都应该遵循的规则:
  • 版权:头部应当添加版权声明注释块；
  • 版权所有:杭州火图科技有限公司

  • 多余空字符:应当被移除，Eclipse用户应当安装 AnyEdit Eclipse Plugin。
  • 缩进:绝对不使用tab键
  • 分行:一行最多120字符
注意：下面的风格详情可能会覆盖通用规则。

Java

Java代码风格

maven项目主要是Java项目，所以maven规则也主要应用于Java。
  • 空格:在参数中间，或者控制关键字之后应当添加一个空格；比如用if ( foo ) 代替 if(foo)，用myFunc( foo, bar, baz )代替myFunc(foo,bar,baz)。方法名后不应该存在空格，比如void myMethod(), myMethod( "foo" )。
  • 缩进:使用4个空格缩进，绝对不用tab键
  • 代码块:总是在新行中用大括号包裹
  • 分行:一行最多120字符，无论是代码还是注释
几个主流IDE的代码风格文件，强烈建议采纳。

IntelliJ IDEA 4.5+

下载 maven-idea-codestyle.xml到~/Library/Preferences/IntelliJIdea14/codestyles然后重启IDE,在首选项-Editor-Code Style中选择maven shcema。
windows中路径为~\.IntelliJIdea14\config\codestyles\

Eclipse 3.2+

下载maven-eclipse-codestyle.xml然后在window-Preferences配置java-code style-Code Formatter 中导入并应用该xml文件。

XML

XML代码风格

  • 缩进:使用2个空格缩进
  • 换行:一个复杂XML数据需要单独分行，简单到一行描述而不可能产生歧义的XML数据无需分行；XML数据类型改变时，需要用一个空行分隔。一个XML示范:
  • <aTag>
      <simpleType>This is a simple type</simpleType>
     
      <complexType>
        <simpleType>This is a complex type</simpleType>
      </complexType>
    </aTag>
单行注释或者注释块都是允许的，比如
 <!-- Simple XML documentation                                               -->
或者
<!-- ====================================================================== -->
<!-- Block documentation                                                    -->
<!-- ====================================================================== -->

POM编码规范

POM中各项定义强烈建议按照以下排序编写。
<project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
  <modelVersion/>
 
  <parent/>
 
  <groupId/>
  <artifactId/>
  <version/>
  <packaging/>
 
  <name/>
  <description/>
  <url/>
  <inceptionYear/>
  <organization/>
  <licenses/>
 
  <developers/>
  <contributors/>
 
  <mailingLists/>
 
  <prerequisites/>
 
  <modules/>
 
  <scm/>
  <issueManagement/>
  <ciManagement/>
  <distributionManagement/>
 
  <properties/>
 
  <dependencyManagement/>
  <dependencies/>
 
  <repositories/>
  <pluginRepositories/>
 
  <build/>
 
  <reporting/>
 
  <profiles/>
</project>
几点说明:
  • <project>元素肯定在新行开启
  • 为了提高可读性使用空行归档了不同类型的定义
  • 没有明文规定在dependencyManagement和dependencies中各个dependency的排序要求，但强烈建议按照用途区分排序（如同我们对整个POM所做的一样）。
